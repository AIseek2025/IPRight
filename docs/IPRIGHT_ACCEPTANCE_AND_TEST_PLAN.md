# IPRight 验收标准与测试设计

## 1. 目标

本文件定义 `IPRight` 的：

- 产品验收标准
- 阶段验收标准
- 自动化测试建议
- 人工抽检要点

用于保证后续 `CodeMaster` 开发完成后，平台不是“功能看起来存在”，而是真正能交付两份软著材料。

## 2. 总体验收标准

当且仅当以下全部满足时，`IPRight` 首版可视为验收通过：

1. 用户可创建任务
2. 平台能生成 PRD 与开发任务书
3. 平台能生成符合契约的标准应用
4. 平台能自动运行应用并通过健康检查
5. 平台能自动截图关键页面
6. 平台能生成包含引言、系统设计、详细使用说明、系统架构图的 Word 版说明书
7. 平台能生成 60 页以上真实源码导向的 Word 版源码文档
8. 平台前端可下载这两份文档

## 3. 阶段验收矩阵

| 阶段 | 输入 | 输出 | 验收标准 |
|------|------|------|----------|
| 任务创建 | 关键词等参数 | task | `task_id` 成功创建 |
| PRD 规划 | 任务参数 | PRD、任务书 | 文档结构完整 |
| 应用生成 | PRD、任务书 | 应用目录与 Manifest | 4 个 Manifest 齐全 |
| 构建安装 | 应用目录 | 构建日志 | 依赖安装成功 |
| 自动运行 | run manifest | runtime status | 健康检查通过 |
| 自动截图 | capture manifest | screenshot manifest + png | 关键页面截图齐全 |
| 说明书导出 | 截图与元数据 | manual docx | 图文对应，结构完整 |
| 源码导出 | code index | source code docx | 页数与行数合法 |
| 下载交付 | 导出记录 | 下载链接 | 用户可下载 |

## 4. 说明书验收标准

## 4.1 结构验收

说明书至少包含：

- 封面
- 引言
- 软件概述
- 开发设计 / 系统设计
- 主要功能说明
- 运行环境
- 详细操作步骤
- 页面截图与图注
- 系统架构图

### 4.1.1 引言验收

- 包含开发背景
- 包含开发目的
- 包含适用领域
- 包含目标用户或使用角色

### 4.1.2 系统设计验收

- 包含开发技术说明
- 包含系统架构图
- 包含运行环境 / 适配环境
- 包含技术特性描述
- 包含核心功能 / 主要功能 / 功能元素说明

### 4.1.3 使用说明验收

- 对主要页面的说明足够详细
- 对关键操作步骤有分步描述
- 对输入、按钮、结果反馈有对应说明
- 对典型业务流程有连续说明

## 4.2 一致性验收

- 每张截图必须来自真实运行页面
- 图注与页面内容一致
- 操作步骤描述与页面元素一致
- 不能出现 PRD 中有、页面中没有的功能描述
- 系统架构图必须与真实模块和真实页面结构一致

## 4.3 格式验收

- 页眉包含软件名称和版本号
- 页码连续
- 每页不少于 30 行
- Word 文件可正常打开
- 图示可正常显示

## 5. 源码文档验收标准

## 5.1 内容验收

- 来源于真实项目源代码
- 不混入第三方依赖
- 文件边界清晰
- 代码可读
- 真实业务代码总量必须达到 60 页以上目标

## 5.2 分页验收

- 总体真实代码页数必须大于 60 页
- 只允许输出单份完整源码文档 `source_code_book.docx`
- 文档内容必须覆盖纳入代码索引的全部源代码
- 每页不少于 50 行
- 页码连续

## 5.3 格式验收

- 页眉包含软件名称和版本号
- 使用等宽字体
- Word 文件可正常打开

## 6. 自动化测试建议

## 6.1 后端 API 测试

覆盖：

- 创建任务
- 查询任务详情
- 查询工件列表
- 查询导出列表
- 重试任务
- 取消任务

## 6.2 状态机测试

覆盖：

- 正常串行流转
- 阶段失败回退
- 阶段重试
- 取消任务
- `needs_review` 收口

## 6.3 Manifest 校验测试

覆盖：

- `app_manifest` 缺字段
- `run_manifest` 缺字段
- `capture_manifest` 路由为空
- `code_index_manifest` 未排除依赖目录

## 6.4 运行与健康检查测试

覆盖：

- 前端入口可达
- 后端健康接口可达
- 登录页可打开
- demo 账号可用

## 6.5 截图链测试

覆盖：

- 能按场景截图
- 截图文件存在
- 截图元数据完整
- 场景失败时能记录错误

## 6.6 Word 生成测试

覆盖：

- 说明书 Word 可生成
- 源码 Word 可生成
- 文档里存在预期章节
- 文档里存在预期图片
- 文档里存在系统架构图
- 源码导出元数据可证明总代码页数大于 60

## 7. 人工抽检建议

即使首版自动化通过，也建议人工抽检以下内容：

- 说明书前 5 页
- 说明书中间 3 个截图章节
- 说明书中的系统架构图
- 源码文档前 3 页
- 源码文档后 3 页
- 下载链接是否真实可用

## 8. 样例验收场景

建议至少准备 3 个基准关键词：

1. `智慧仓储管理平台`
2. `设备巡检管理系统`
3. `客户信息管理平台`

这 3 类产品都属于后台管理型系统，适合作为 MVP 验收样本。

## 9. 红线失败标准

以下任一情况，任务应判定为不通过：

- 说明书没有真实截图
- 截图与说明文字不对应
- 说明书缺少引言 / 系统设计 / 使用说明主体模块
- 说明书缺少系统架构图
- 源码文档混入依赖目录
- 源码真实业务代码总量不足 60 页
- 两份文档来自不同构建版本
- 用户无法从前端下载
- 平台只能生成 PRD，不能走完整链

## 10. 给 CodeMaster 的验收建议

后续无人值守开发时，`CodeMaster` 应优先把验收写成结构化产物，例如：

- `acceptance_report.json`
- `manual_validation_report.json`
- `code_book_validation_report.json`

这样平台就能对“是否达标”做自动判定，而不是靠最终人工猜测。
