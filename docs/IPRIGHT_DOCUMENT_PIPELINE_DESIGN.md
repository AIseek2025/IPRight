# IPRight 文档生成流水线设计

## 1. 设计目标

`IPRight` 最终必须稳定产出两份 Word 文档：

1. `软件说明书 / 操作手册`
2. `软件源代码文档`

本设计文件专门说明如何把真实应用、真实截图、真实源代码变成这两份可交付文档。

## 2. 外部规范映射

根据 `软件源代码文档和软件说明书操作手册撰写说明及格式规范.md`，平台必须满足以下约束：

## 2.1 软件源代码文档

- 前后各 30 页，共 60 页
- 如不足 60 页，则提交全部
- 每页不少于 50 行
- 页眉写软件名称与版本号
- 右上角页码连续
- 内容需为可读性强的真实代码

## 2.2 软件说明书 / 操作手册

- 每页不少于 30 行
- 页眉写软件名称与版本号
- 右上角页码连续
- 内容必须系统覆盖引言、开发设计/系统设计、软件使用说明
- 图文与真实软件一致
- 必须包含可视化的系统架构图

## 3. 说明书生成设计

## 3.1 总体流程

```text
生成应用
-> 启动应用
-> 健康检查
-> 登录与页面遍历
-> 截图与页面元数据采集
-> 按截图生成图注和操作说明
-> 按章节模板编排
-> 输出 Word
```

## 3.2 说明书章节建议

建议固定以下章节结构，并作为每一份说明书的默认工作标准：

1. 封面
2. 文档说明
3. 引言
4. 软件概述
5. 开发设计 / 系统设计
6. 软件运行环境 / 适配环境
7. 功能结构说明
8. 软件使用说明
9. 典型业务流程说明
10. 技术特点说明
11. 常见问题

### 3.2.1 引言标准

引言至少包含：

- 开发背景
- 开发目的
- 适用领域
- 目标用户 / 使用角色
- 建设价值与应用场景

### 3.2.2 开发设计 / 系统设计标准

开发设计 / 系统设计章节至少包含：

- 开发技术说明
- 系统架构图
- 开发运行环境 / 软件适配环境
- 技术特性描述
- 软件核心功能 / 软件主要功能 / 功能元素说明

### 3.2.3 软件使用说明标准

软件使用说明应尽量详细，至少覆盖：

- 系统登录 / 进入方式
- 首页 / 仪表盘说明
- 各主功能页面的操作路径
- 关键按钮、输入项、结果反馈说明
- 典型业务流程的分步操作
- 异常提示、退出、常见问题处理

## 3.3 截图策略

### 3.3.1 不是“截所有 DOM 页面”

说明书真正需要的是“有业务价值的页面证据”，因此截图应以场景为中心，而不是只按路由数量。

### 3.3.2 推荐截图层级

每个应用最少保证以下截图：

- 登录页
- 首页 / 仪表盘
- 列表页
- 新增表单页
- 详情页
- 编辑页
- 系统设置页
- 关键流程完成态页面

### 3.3.3 截图元数据结构

```json
{
  "scenario_id": "user-list",
  "route": "/users",
  "page_title": "用户管理",
  "image_file": "005_user_list.png",
  "preconditions": ["login_as_admin"],
  "actions": ["open_user_menu"],
  "visible_labels": ["新增用户", "搜索", "状态"],
  "suggested_caption": "图5 用户管理页面"
}
```

## 3.4 页面说明文字生成

### 输入

- 页面标题
- 路由
- 可见控件标签
- 所属模块
- 截图上下文

### 输出

- 图注
- 50-120 字的页面说明
- 3-6 步的操作步骤

### 文案要求

- 正式、客观
- 不编造不存在功能
- 尽量使用页面上真实可见的字段与按钮名称

## 3.5 说明书 Word 模板设计

建议模板组成：

### 封面区

- 软件名称
- 版本号
- 文档名称：软件说明书 / 操作手册
- 日期

### 正文区

- 自动目录
- 分节标题
- 架构图
- 页面截图
- 图注
- 操作说明

### 页眉页脚

- 页眉：`{软件名称}{版本号} 说明书`
- 页码：右上角

## 3.6 说明书页数与行数策略

说明书需要满足每页不少于 30 行。为此建议采用：

- 章节正文使用常规段落
- 每张截图不独占整页，而是“截图 + 图注 + 说明文字”
- 长文段与截图交叉编排
- 当截图较多时，自动缩放图片，避免文字行数不足

## 3.7 图示生成策略

### 3.7.1 必须生成的图示

每份说明书至少生成以下图示：

- 系统架构图

### 3.7.2 图示生成实现建议

推荐实现路线：

1. 平台先产出结构化图示描述，例如 `diagram_spec.json`
2. 再使用 `Mermaid`、`Graphviz` 或 `PlantUML` 之一生成 `svg/png`
3. 将渲染后的图片插入 Word 文档

### 3.7.3 图示结构化输入建议

```json
{
  "architecture": {
    "nodes": ["前端", "后端 API", "任务编排器", "数据库", "对象存储"],
    "edges": [["前端", "后端 API"], ["后端 API", "数据库"], ["任务编排器", "对象存储"]]
  },
  "notes": ["图示仅描述软件系统结构，不描述生成流程"]
}
```

### 3.7.4 生成原则

- 图示必须与真实产品模块一致
- 不允许使用与实际实现无关的通用模板图替代
- 图示与文字描述必须互相对应
- 不再生成开发设计流程图或自动开发流程图

## 4. 源代码文档生成设计

## 4.1 代码来源选择原则

源码文档应优先收录：

- 前端页面代码
- 前端状态管理与路由代码
- 后端接口与核心服务代码
- 数据模型

应排除：

- 构建产物
- 第三方依赖
- 自动生成 lock 文件
- 压缩文件
- 大体量无代表性资源文件

## 4.2 源码合并策略

平台不直接按目录拼接，而是：

1. 读取 `code_index_manifest.json`
2. 按优先级排序文件
3. 输出带文件边界标记的“规范化源码流”

示例：

```text
===== FILE: frontend/src/App.tsx =====
...代码...
===== FILE: frontend/src/pages/Dashboard.tsx =====
...代码...
```

## 4.2.1 总码量目标

从平台工作标准看，生成应用不能只满足“能运行”，还必须满足软著源码文档的可交付性：

- 真实业务代码总页数目标应大于 60 页
- 如当前总代码页数不足 60，视为功能体量仍然不足，应继续扩展产品模块
- 扩展方向优先增加真实业务页面、真实后端接口、真实服务层与真实数据模型，不允许通过重复样板或无意义注释凑页

## 4.3 分页算法

### 约束

- 每页至少 50 行
- 优先每页固定 55 行，兼顾版面余量

### 处理步骤

1. 将规范化源码流按行切分
2. 补齐文件路径标记行
3. 每 55 行切一页
4. 如果总页数 `<= 60`，说明产品体量未达标，应继续扩展功能后再导出正式版
5. 如果总页数 `> 60`：
   - 导出单份完整版本 `source_code_book.docx`
   - 在导出元数据中明确该文档覆盖全部纳入索引的源码内容

### 关键点

- 不允许为了凑页数插入无关内容
- 可保留注释，但注释不应替代真实业务代码
- 文件边界标记要算入行数

## 4.4 源码 Word 模板

### 格式要求

- A4
- 等宽字体，优先 `Consolas`
- 字号 `9-10.5`
- 单倍或固定行距
- 页眉：`{软件名称}{版本号} 源代码`
- 页码右上角

### 页面结构

- 页眉
- 页码
- 代码正文

## 4.5 不足 60 页时的处理

若总代码页数不足 60 页：

- 不应直接视为达标
- 应优先回到产品生成阶段补充真实功能与真实代码
- 禁止通过空白页、无意义注释、重复代码拼接来凑页
- 在验收中记录为 `code_volume_insufficient`

## 5. 自动运行与截图的关键技术点

## 5.1 必须有标准启动契约

每个生成应用必须提供：

- 安装命令
- 启动命令
- 端口信息
- 健康检查 URL
- Demo 账号

否则无法稳定进入截图阶段。

## 5.2 必须有标准截图契约

每个生成应用必须提供：

- 页面清单
- 登录前置条件
- 动作清单
- 截图标题
- 截图优先级

否则截图会遗漏关键页面，说明书也会失真。

## 5.3 必须有标准代码索引契约

每个生成应用必须提供：

- 纳入源码文档的文件集合
- 排除文件集合
- 文件排序建议

否则完整源码文档会混入低价值代码。

## 6. 失败场景与兜底策略

## 6.1 应用启动失败

### 表现

- 服务端口未监听
- 健康检查失败

### 兜底

- 自动修复一次配置
- 记录启动日志
- 进入 `needs_review`

## 6.2 登录失败

### 表现

- 页面未进入首页
- Demo 账号无效

### 兜底

- 使用预设账号重试
- 如果失败则跳过需要登录的截图，但保留错误记录

## 6.3 页面截图失败

### 表现

- 路由无法访问
- 页面持续 loading

### 兜底

- 再试一次
- 降级抓取页面可见区域
- 记录为“缺图场景”，不直接静默跳过

## 6.4 Word 编排失败

### 表现

- 图片插入失败
- 样式损坏

### 兜底

- 保留 JSON 中间结果
- 重跑文档编排阶段，不重新开发

- `software_manual.docx`

## 7.2 源码文档相关

- `source_code_book.docx`
- `source_code_render.json`

- `normalized_code_stream.txt`
- `diagram_spec.json`

## 8. 关键产品判断

最重要的不是“让模型会写 Word”，而是让平台形成如下稳定工艺：

1. 真实运行
2. 真实截图
3. 真实源码索引
4. 模板化编排
5. 结构化导出

只有这五步都被平台产品化，`IPRight` 才能稳定生成可交付的软著材料。
