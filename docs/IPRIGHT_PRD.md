# IPRight 平台 PRD

## 1. 产品概述

### 1.1 产品名称

`IPRight`

### 1.2 产品使命

将“关键词 -> 软件 -> 截图 -> 说明书 Word -> 源码 Word -> 下载交付”串成一条无人值守的软著材料生成流水线。

### 1.3 目标问题

传统软著材料交付存在四类高频痛点：

1. 软件说明书依赖人工截图和人工写步骤
2. 源码文档需要人工挑选前后页并排版
3. 软件开发和文档生产脱节，导致图文不一致
4. 最终交付缺乏可重复、可追溯的流水线

## 2. 用户场景

### 2.1 场景 A：方案型客户

- 用户输入关键词：`智慧园区管理平台`
- 平台自动产出 PRD、开发计划、可运行 Demo、说明书 Word、源码 Word
- 用户下载后直接用于软著材料整理

### 2.2 场景 B：批量交付型服务商

- 运营人员批量创建多个任务
- 平台自动跑多个项目
- 每个任务生成两份 Word 文档并归档

### 2.3 场景 C：文档返工

- 用户对说明书不满意
- 平台复用已有代码与截图资产重新生成说明书
- 避免从头再开发

## 3. 用户旅程

1. 用户输入关键词
2. 用户补充可选参数
   - 软件名称
   - 版本号
   - 目标行业
   - 风格偏好
3. 平台生成 PRD 与开发任务书
4. 平台生成并开发目标软件
5. 平台运行软件并自动验证
6. 平台自动截图页面
7. 平台生成说明书 Word
8. 平台生成源码 Word
9. 平台将两份文档提供下载

## 4. 功能模块

## 4.1 任务创建模块

### 输入项

- 关键词
- 软件名称
- 版本号
- 业务类型
- 可选补充描述

### 输出项

- `task_id`
- 任务状态：`queued`
- 初始任务记录

### 核心要求

- 支持最小输入启动任务
- 支持后续追踪任务状态

## 4.2 PRD 与开发任务书生成模块

### 输出文档

- `product_prd.md`
- `development_work_order.md`
- `route_plan.json`
- `data_model.json`

### PRD 必须覆盖

- 产品定位
- 用户角色
- 页面结构
- 业务流程
- 数据实体
- 核心功能模块
- 非功能要求

### 开发任务书必须覆盖

- 技术栈
- 页面任务拆解
- API 任务拆解
- 数据模型
- 测试要求
- 运行要求
- 截图要求
- 文档生成要求

## 4.3 自动开发模块

### 目标

根据 PRD 与任务书自动生成一个可运行的软件应用。

### MVP 范围内约束

- 首期固定支持 `后台管理型 Web 应用`
- 首期固定支持标准技术模板
- 首期要求生成应用输出标准 Manifest

### 必须产出的开发工件

- `app_manifest.json`
- `run_manifest.json`
- `capture_manifest.json`
- `code_index_manifest.json`
- `README_RUN.md`

## 4.4 自动运行与健康检查模块

### 目标

让平台在无人工介入时成功启动生成的软件，并判断是否可以进入截图阶段。

### 核心功能

- 安装依赖
- 启动服务
- 检查端口与健康接口
- 检查登录页与主页面可达性
- 记录运行日志

### 输出

- `runtime_status.json`
- `health_report.json`
- `launch_log.txt`

## 4.5 页面截图模块

### 目标

自动遍历应用中的关键页面，并为每个页面生成至少一张可写入说明书的截图。

### 页面来源

- `capture_manifest.json`
- 路由配置
- 页面元素探测结果

### 每张截图的元数据

- 截图文件名
- 页面名称
- 所属功能模块
- 路由
- 前置动作
- 截图说明上下文
- 关键可见元素
- 推荐图注

### 输出

- `/artifacts/screenshots/*.png`
- `screenshot_manifest.json`

## 4.6 软件说明书 Word 生成模块

### 目标

根据真实运行软件与真实截图，生成符合软著要求的 Word 版说明书 / 操作手册。

### 必须包含的章节

- 封面
- 文档修订记录
- 软件概述
- 产品功能说明
- 技术特点
- 运行环境
- 操作流程
- 页面操作说明
- 常见问题

### 关键规则

- 每一张截图都必须对应说明文字
- 说明必须引用真实页面名称与真实操作动作
- 内容应与生成的软件一致

### 输出

- `software_manual.docx`
- `software_manual_render.json`

## 4.7 源代码 Word 生成模块

### 目标

将核心源代码按照软著规范导出为 Word 文档。

### 关键规则

- 只生成单份完整源码文档 `source_code_book.docx`
- 文档必须覆盖纳入代码索引的全部源代码
- 每页不少于 50 行
- 如果总代码页数不足 60 页，则说明产品体量不足，应继续补充真实功能
- 页眉包含软件名称与版本号
- 页码连续

### 输出

- `source_code_book.docx`
- `source_code_render.json`

## 4.8 下载与交付模块

### 前端展示

- PRD 生成状态
- 开发状态
- 截图状态
- 文档生成状态
- 最终下载区

### 可下载文件

- 软件说明书 Word
- 源代码 Word
- 可选：PRD、开发计划、截图包

## 6. 状态机设计

任务状态建议定义为：

1. `queued`
2. `planning`
3. `coding`
4. `building`
5. `running`
6. `capturing`
7. `writing_manual`
8. `writing_code_book`
9. `publishing`
10. `completed`
11. `failed`
12. `needs_review`

## 7. 非功能需求

### 7.1 可观测性

- 所有阶段必须有结构化日志
- 所有工件必须落盘
- 失败必须有失败原因

### 7.2 可恢复性

- 文档生成失败可单独重试
- 截图失败可单独重试
- 开发成功后的说明书重写不应重复开发

### 7.3 安全性

- 生成软件必须在沙箱运行
- 平台与生成应用隔离
- 上传下载文件做病毒扫描和访问控制

### 7.4 一致性

- 文档版本、截图版本、代码版本必须绑定同一构建 ID

## 8. 关键产品决策

### 决策 1：限制产品类型

MVP 先只支持“后台管理型 / CRUD 型 / 仪表盘型 Web 应用”，不直接支持复杂 3D、实时游戏、强交互可视化项目。

### 决策 2：固定开发脚手架

为了保证自动运行与自动截图稳定，生成应用不允许完全自由发挥，必须基于标准脚手架。

### 决策 3：截图优先走场景清单

平台优先根据 `capture_manifest.json` 执行截图，而不是全站盲爬。

### 决策 4：说明书基于截图资产生成

说明书不是独立写作，而是“真实截图资产 + LLM 文案生成 + 模板编排”的组合结果。

## 9. 验收标准

- 输入关键词后可以创建任务
- 能生成 PRD 与开发任务书
- 能产出可运行软件
- 能自动拿到截图资产
- 能导出符合规范的 `software_manual.docx`
- 能导出符合规范的 `source_code_book.docx`
- 前端可下载这两份文档
