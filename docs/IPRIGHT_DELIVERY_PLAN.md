# IPRight 建设路线与交付计划

## 1. 建设策略

`IPRight` 不适合一开始追求“支持任意软件类型”。  
正确路线是：

1. 先支持一类最容易标准化的应用
2. 先保证两份 Word 文档稳定交付
3. 再逐步扩展复杂页面与复杂技术栈

## 2. MVP 范围

MVP 版本只支持：

- 后台管理型 Web 应用
- 标准登录页 / 首页 / 列表页 / 表单页 / 详情页
- 固定脚手架生成
- 固定的说明书模板
- 固定的源码文档模板

## 3. 分阶段里程碑

## Phase 0：方案冻结

### 目标

- 冻结产品范围
- 冻结技术栈
- 冻结两份 Word 的模板规范

### 输出

- 产品蓝图
- PRD
- 技术架构
- 文档生成设计

## Phase 1：平台骨架

### 目标

- 建立任务系统
- 建立状态机
- 建立对象存储
- 建立下载能力

### 输出

- 任务创建页
- 任务详情页
- Task API
- Artifact API

## Phase 2：PRD 与开发任务书生成

### 目标

- 根据关键词自动生成 PRD
- 生成开发任务书和页面清单

### 验收

- 任务创建后能产出完整 PRD
- PRD 能稳定约束到标准后台管理型产品

## Phase 3：标准应用生成

### 目标

- 让编程模型输出符合 `IPRight App Contract` 的应用
- 生成 `run_manifest`、`capture_manifest`、`code_index_manifest`

### 验收

- 生成应用可本地启动
- Manifest 信息完整

## Phase 4：运行与截图流水线

### 目标

- 自动安装依赖
- 自动启动服务
- 自动截图
- 自动记录截图元数据

### 验收

- 能稳定抓到登录页、首页、列表页、表单页

## Phase 5：说明书 Word 生成

### 目标

- 将截图与说明文字编排进 Word
- 满足页眉、页码、行数、章节结构要求

### 验收

- 说明书可打开
- 说明图文对应
- 页面说明与真实 UI 一致

## Phase 6：源码 Word 生成

### 目标

- 规范化源码流
- 分页
- 单份完整源码文档导出

### 验收

- 每页不少于 50 行
- 页码连续
- 代码可读

## Phase 7：交付与运营

### 目标

- 文档下载
- 重试机制
- 审核机制
- 基础监控

### 验收

- 用户可直接下载两份 Word
- 失败任务可重试

## 4. 推荐开发顺序

### 第一优先级

- 任务系统
- PRD 生成
- 标准应用脚手架
- 截图流水线

### 第二优先级

- 说明书 Word 编排
- 源码 Word 编排
- 下载页

### 第三优先级

- 审核台
- 失败重试
- 统计看板

## 5. 核心风险与缓解

## 风险 1：生成软件不可运行

### 缓解

- 只支持标准脚手架
- 运行前做静态检查
- 必须产出运行契约

## 风险 2：截图覆盖不完整

### 缓解

- 强制产出截图场景清单
- 对核心页面设最小覆盖要求

## 风险 3：说明书文字与页面不一致

### 缓解

- 文案必须基于截图元数据生成
- 引用页面真实标签

## 风险 4：源码文档质量差

### 缓解

- 强制产出源码索引
- 禁止把第三方依赖混入正文

## 风险 5：异步长任务不稳定

### 缓解

- 任务拆阶段
- 每阶段工件落盘
- 支持从阶段重试

## 6. 团队分工建议

### 产品

- 负责产品模板定义
- 负责 PRD Prompt 与说明书模板

### 平台后端

- 负责任务系统
- 负责状态机与对象存储
- 负责导出下载

### AI 工程

- 负责 PRD 生成链
- 负责代码生成契约
- 负责说明书文案生成

### Agent / 自动化

- 负责沙箱运行
- 负责 Playwright 截图
- 负责健康检查和动作编排

### 文档引擎

- 负责 Word 模板和编排逻辑
- 负责源码分页与导出

## 7. 首版验收标准

首版不要求覆盖所有软件类型，但必须完成：

- 一键创建任务
- 一键生成后台管理型 Web 软件
- 自动运行并截图
- 自动生成一份带截图说明书 Word
- 自动生成一份完整源码 Word
- 前端可下载

## 8. 最终建议

`IPRight` 首期最重要的不是“大而全”，而是把这条链打穿：

`关键词 -> PRD -> 标准应用 -> 运行 -> 截图 -> 说明书 Word -> 源码 Word -> 下载`

只要这条链稳定跑通，后续才值得继续扩展：

- 更多应用模板
- 更多行业领域
- 更多导出格式
- 更强的自动修复能力
