# IPRight 最佳运行流程与迭代升级方案

## 1. 文档目的

本文件用于沉淀 `IPRight` 当前最适合真实生产运行的端到端流程，作为后续长期维护、修复、重构和升级时的统一参照。

本文件不是理想化蓝图，而是基于当前代码实现、生产任务运行结果和最近多轮失败复盘，总结出的“现阶段最优可执行方案”。

最近一次真实生产端到端打通样例见：

- `docs/archive/IPRIGHT_PRODUCTION_SUCCESS_CHAIN_2026-05-22.md`

适用范围：

- 任务从创建到最终导出全链路
- `PRD`、代码生成、运行验证、截图、说明书、源码文档
- 生产环境迭代修复时的判断基线
- 后续升级时的流程评估标准

## 2. 核心结论

当前最合适的总体策略是：

1. 先由 `deepseek-v4-flash` 生成 `PRD`、开发任务书和产品摘要
2. 再由 `deepseek-v4-pro` 基于 `PRD + 开发任务书 + 任务画像` 生成应用代码
3. 在运行环境中启动前端和后端，先做 `verify_run`
4. 仅在运行验证通过后再做页面截图
5. 再由 `deepseek-v4-flash` 基于 `PRD 摘要 + 项目画像 + 截图元数据` 生成说明书正文素材、图注和页面说明
6. 最后由程序化文档生成器统一插入截图、架构图和章节内容，导出 `software_manual.docx`

该方案优于“先写说明书再补截图”或“只运行前端就截图”的原因在于：

- `PRD` 更适合由文本模型快速整理结构与业务范围
- 代码生成更依赖复杂约束遵循与多轮回补，适合由推理模型主导
- 截图必须基于真实运行页面，不能基于静态页面或想象页面
- 说明书正文既依赖截图，也依赖 `PRD`、模块画像和技术画像，不能只靠截图孤立撰写
- 最终文档排版必须由程序控制，避免模型直接输出不可控的 Word 结构

## 3. 最佳模型分工

### 3.1 主分工

- `deepseek-v4-flash`
  - 负责 `PRD`、开发任务书、说明书正文素材、图注、页面说明
  - 优点是中文长文组织速度快、成本低、适合结构化文档输出

- `deepseek-v4-pro`
  - 负责应用代码生成、复杂回补和约束更强的代码修复
  - 优点是更适合多文件代码生成、细粒度约束执行和复杂重试场景

### 3.2 例外分工

- 单文件代码回补时，如果 `deepseek-v4-pro` 返回空内容，可允许 `deepseek-v4-flash` 做一次纯文本源码兜底重试
- 这只是稳定性补偿，不改变“代码主模型是 `v4-pro`”这一总体原则

### 3.3 长期原则

- 文档类任务优先使用 `flash`
- 代码类任务优先使用 `pro`
- 不要为了统一模型而牺牲任务匹配度
- 若未来模型能力变化，应按真实成功率而不是按名称预设更新分工

## 4. 最佳端到端流程图

```text
任务创建
-> Plan
   -> v4-flash 生成 PRD / 开发任务书 / prd_summary
-> Build
   -> 基于 PRD 摘要构建 project_profile
   -> v4-pro 生成前后端代码与 manifests
   -> 无效文件回补 / 单页回补 / 单文件 plaintext 回补
-> Verify Run
   -> 安装依赖
   -> 启动 frontend preview + backend api
   -> 健康检查
-> Capture
   -> 根据 capture_manifest 打开真实页面
   -> 逐页截图并写入 screenshot_manifest
-> Compose Manual
   -> v4-flash 生成说明书正文素材 / 图注 / 页面说明
   -> 程序插入截图、架构图并输出 software_manual.docx
-> Compose Code Book
   -> 程序输出 source_code_book.docx
-> Publish
   -> 发布说明书、源码文档、申请表和整包下载
```

### 4.2 模型与程序职责总表

| 环节 | 主执行方 | 主要职责 | 不负责的内容 |
| --- | --- | --- | --- |
| Plan | `v4-flash` | 生成 `PRD`、开发任务书、`prd_summary` | 不直接写应用代码 |
| Build | `v4-pro` | 生成前后端代码、完成多轮代码回补 | 不直接生成最终说明书 |
| Verify Run | 程序 | 安装依赖、启动服务、健康检查 | 不改写业务文案 |
| Capture | 程序 | 进入真实页面并截图、生成截图清单 | 不臆造页面说明 |
| Compose Manual | `v4-flash` + 程序 | 生成说明书正文素材、图注、页面说明，并排版导出 | 不跳过真实截图直接造文档 |
| Compose Code Book | 程序 | 生成源码文档 | 不决定业务主题 |
| Publish | 程序 | 注册导出物与下载入口 | 不修改正文内容 |

## 5. 分阶段最佳方案

### 5.1 Plan 阶段

目标：

- 生成任务专属 `PRD`
- 生成开发任务书
- 生成 `prd_summary`

输入：

- 用户原始需求
- 产品名、关键词、版本号、行业、备注
- 平台生成的 `plan_seed`

输出：

- `prd/product_prd.md`
- `prd/development_work_order.md`
- `prd/product_summary.json`

最佳实践：

- `PRD` 必须以原始用户输入为唯一主题源
- 平台辅助画像只能补充结构，不能替代产品主题
- `PRD` 必须显式输出模块、角色、页面路由、场景主线
- `PRD` 失败时直接失败，不做模板化 `PRD` 兜底

### 5.2 Build 阶段

目标：

- 生成真实可运行应用
- 生成 `app_manifest`、`run_manifest`、`capture_manifest`、`code_index_manifest`
- 生成 `project_profile.json` 和 `app_codegen_report.json`

输入：

- `PRD`
- 开发任务书
- `prd_summary`
- 基于任务构建出的 `project_profile`

输出：

- 前后端应用代码
- 各类 manifest
- 代码生成报告

最佳实践：

- `deepseek-v4-pro` 作为代码主模型
- 核心页面与模块页面都必须由模型真实生成
- 平台禁止对 `App.tsx`、`Login.tsx`、`Dashboard.tsx` 或任意模块页做产品界面模板兜底
- 无效模块页必须走回补链路，不得静默跳过
- 模块页回补优先单文件、低温度、纯文本协议，降低 JSON 截断概率
- 任何阶段都必须过滤越界文件，只接收当前 `required_files`

### 5.3 Verify Run 阶段

目标：

- 验证生成应用真实可运行

输入：

- `run_manifest`
- `app_manifest`
- 工作区代码

输出：

- 运行状态
- 健康检查结果

最佳实践：

- 必须同时启动前端和后端，不建议只运行前端
- 构建失败、依赖失败、健康检查失败都必须明确拦截
- `verify_run` 必须在 `capture` 之前

### 5.4 Capture 阶段

目标：

- 抓取真实页面截图
- 输出截图清单

输入：

- `capture_manifest`
- `run_manifest`
- 实际运行中的应用

输出：

- `screenshots/*.png`
- `screenshot_manifest.json`

最佳实践：

- 每次 build 必须重新截图，禁止复用上一轮截图
- 关键页面缺失时直接判失败
- 截图必须面向业务页面，不做无意义整页采样
- `capture` 完成后才能进入说明书生成

### 5.5 Compose Manual 阶段

目标：

- 生成企业口径的软件说明书

输入：

- `project_profile`
- `prd_summary`
- `screenshot_manifest`
- 实际截图文件

输出：

- `software_manual.docx`
- 说明书相关中间内容

最佳实践：

- `deepseek-v4-flash` 负责说明书正文素材、页面说明、图注
- 程序负责 Word 排版、截图插入和架构图插入
- 正文必须以企业身份陈述真实产品设计与功能事实
- 不得在说明书中暴露模型、自动生成、截图采集流程
- 不得使用截图导向措辞，如“该截图展示了”

### 5.6 Compose Code Book / Publish 阶段

目标：

- 导出源码文档、申请表和整包交付物

最佳实践：

- `source_code_book.docx` 与说明书独立生成
- 发布前再核对产品名、版本号、截图数量和导出索引

## 6. 阶段输入输出责任表

| 阶段 | 关键输入 | 关键输出 | 通过条件 | 失败收口 |
| --- | --- | --- | --- | --- |
| Plan | 原始需求、产品名、关键词、行业、备注、`plan_seed` | `product_prd.md`、`development_work_order.md`、`product_summary.json` | `PRD` 结构完整、摘要可用于下游画像 | 直接失败，不允许模板 PRD 兜底 |
| Build | `PRD`、任务书、`prd_summary`、`project_profile` | 应用代码、四类 manifest、`project_profile.json`、`app_codegen_report.json` | 关键代码文件齐全、校验通过、无无效模块页残留 | 进入回补；仍失败则 build 失败 |
| Verify Run | `run_manifest`、应用代码 | 运行状态、健康检查结果 | 前后端均可启动并通过检查 | 直接失败，禁止跳到截图 |
| Capture | `capture_manifest`、运行中的应用 | 截图文件、`screenshot_manifest.json` | 核心页面截图完成，标题完整 | 直接失败，禁止复用旧截图 |
| Compose Manual | `project_profile`、`prd_summary`、截图元数据、截图文件 | `software_manual.docx`、`application_form.docx` | 图文一致、章节完整、企业口吻合规 | 直接失败或进入文档修复 |
| Compose Code Book | `code_index_manifest`、源码目录 | `source_code_book.docx` | 目录与页数规则满足要求 | 直接失败 |
| Publish | 全部导出物 | 下载入口、整包导出 | 导出记录齐全 | 直接失败 |

## 7. 为什么当前最优流程必须这样设计

最近真实运行暴露出几个明确结论：

### 6.1 不能让平台兜底写产品页面

如果平台用固定后台模板兜底，虽然短期更容易“跑通”，但会带来三个严重问题：

- 产物与任务主题不匹配
- 页面高度同质化
- 说明书与真实业务截图失真

因此平台只能做：

- 规则约束
- 回补调度
- 校验
- 失败收口

平台不能代替模型做最终产品设计。

### 6.2 不能先写说明书再截图

如果先写说明书，后截图，经常会出现：

- 正文讲了截图里没有的内容
- 截图里有真实功能，但说明书没覆盖
- 页面标题、图注、正文描述互相错位

因此正确顺序必须是：

`真实运行 -> 真实截图 -> 基于截图和画像写说明书`

### 6.3 不能只依赖截图写说明书

截图能说明页面长什么样，但说明不了完整的：

- 开发背景
- 技术架构
- 模块协同
- 运行环境
- 角色权限
- 软件总体业务流程

因此说明书必须同时使用：

- `PRD 摘要`
- `project_profile`
- `screenshot_manifest`
- 页面截图

### 6.4 当前真正难点在 Build 的最后一公里

最近连续多轮失败并不说明整个流程错了，而是说明：

- 主流程已经基本正确
- 最难的瓶颈集中在 `module_invalid_retry`
- 少数页面会回退到历史主题或通用壳页，导致 build 在最后阶段失败

因此当前应优先优化的是“代码生成稳定性”，不是推翻整个流程顺序。

## 8. 常见失败模式与对应优化动作

| 失败模式 | 常见表现 | 优先优化动作 | 不建议的错误做法 |
| --- | --- | --- | --- |
| `LLM JSON parse error` | `finish_reason=length`、返回 JSON 截断 | 缩小 batch、单文件纯文本协议、降低单次输出体积 | 继续扩大批次或盲目加 token |
| `empty code body` | 单页回补返回空文本 | 同族模型二次重试、补充“从第一行开始输出源码”强约束 | 直接判失败不重试 |
| 半截 TSX | `Unterminated string literal`、括号不闭合 | build 阶段完整性校验前移 | 等到 `verify_run` 再发现 |
| 历史主题漂移 | 页面标题变成“候选人管理”“职位管理”等旧主题 | 把真实坏标题写回负例约束和 `validation_hints` | 放宽校验让错误页面过关 |
| 越界文件污染 | 单页回补返回了 `App.tsx` 等非当前文件 | 接收端严格过滤 `required_files` | 默认接受模型多返回的文件 |
| 模板化壳层回退 | 通用后台壳页、通用产品壳页反复出现 | 禁止平台 UI 兜底、加强标题和业务锚点校验 | 用平台模板把 build 伪装成成功 |
| 截图串图 | 旧截图混入新任务 | 每轮重置截图目录、重建截图清单 | 复用上一轮截图 |
| stale build 悬挂 | dashboard 显示 running 但 worker 已死 | 清理 stale build 状态、恢复 active_build_id 一致性 | 反复重试同一悬挂 build |

## 9. 当前阶段的重点优化方向

### 7.1 第一优先级：稳住模块页回补

目标：

- 不再让残留失败页在 `RecordsPage`、`WorkflowPage`、`AssetsPage` 之间来回迁移

做法：

- 持续把线上真实坏输出写回负例约束
- 强化模块标题逐字命中校验
- 强化业务锚点命中校验
- 持续保持单文件 plaintext 回补

### 7.2 第二优先级：让失败更早暴露

目标：

- 不把半截源码、错主题页面拖到后续阶段才发现

做法：

- 在 build 阶段加强源码完整性校验
- 在 build 阶段加强页面主题校验
- 在截图前确保关键页面标题和必要元素存在

### 7.3 第三优先级：增强说明书最终审阅

建议新增：

- 说明书正文和页面说明生成后，增加一次统一审阅环节
- 该环节可使用更强模型做一致性审核，但不直接负责主生成

审核重点：

- 图注与页面标题是否一致
- 术语是否统一
- 是否有截图导向措辞
- 是否暴露模型或自动化流程
- 是否出现历史任务残留文本

## 10. 长期升级机制

本文件应视为持续更新的长期基线文档。后续每次遇到新的真实问题，都应按以下方式升级：

1. 先确认问题属于哪个阶段
2. 只修对应阶段，不轻易推翻整体流程
3. 先记录“真实坏输出”或“真实失败模式”
4. 再决定是修 prompt、修校验、修回补协议，还是修阶段顺序
5. 每次稳定收敛后，把新经验写回本文件

## 11. 文档更新规则

以下情况必须更新本文件：

- 模型主分工发生变化
- 阶段顺序发生变化
- 新增关键阶段或关键工件
- 真实运行中发现旧流程不再适合
- 说明书或源码文档生成机制出现系统级变化

更新要求：

- 必须写清“变更前 -> 变更后 -> 变更原因”
- 必须说明对生产任务成功率和文档质量的影响
- 必须同步相关主文档与实施代码

## 12. 当前正式建议

截至当前版本，`IPRight` 的正式建议流程为：

1. `v4-flash` 生成 `PRD / 开发任务书`
2. `v4-pro` 生成应用代码
3. 运行前后端并完成 `verify_run`
4. 基于真实运行页面完成截图
5. `v4-flash` 生成说明书正文素材与页面说明
6. 由程序生成器合成最终说明书和源码文档
7. 在真实失败中持续反哺规则、校验和回补策略

这是当前最贴近真实生产需求、最利于长期迭代、也最符合软著说明书真实性要求的最佳方案。

## 13. 后续维护建议

为避免后续再次陷入“口头讨论很多、基线不断漂移”的状态，建议固定以下维护方式：

1. 流程级变更先改本文件，再改实现代码
2. 每次系统级修复必须补一条“失败模式 -> 优化动作”映射
3. 每次模型分工调整必须补成功率依据，不能只凭感觉切模型
4. 每次阶段顺序调整必须说明会影响哪些工件和哪些下游阶段
5. 若后续形成更稳定的新流程，应在本文件中保留“旧流程为何被替换”的说明，避免重复走回头路
