# IPRight 文档生成制作规范

## 1. 适用范围

本规范用于约束 `IPRight` 后续每一次生成的软件说明书 / 操作手册、源代码文档及其相关导出物。

所有后续生成任务均必须遵循本规范，不得因单次项目差异而随意放宽。

### 1.1 执行前阅读要求

- 每次开始新的生成、修复或规则调整任务前，必须先尽可能完整阅读当前工作文件夹中的相关内容，对项目形成全面理解后再动手
- 当前项目的基础工作文件夹为 `/Users/brando/Documents/trae_projects/CodeMaster/isolated_autoruns/IPRight`
- 至少必须优先阅读以下材料，并以其作为后续执行基线：
  - `docs/IPRIGHT_CODE_AUDIT_REPORT_2026-05-08.md`
  - `docs/IPRIGHT_RECENT_REPAIR_SUMMARY_2026-05-08.md`
  - `docs/IPRIGHT_DOCUMENT_GENERATION_RULES.md`
  - `docs/IPRIGHT_INDEPENDENT_DELIVERY_CHECKLIST.md`
  - `docs/软件说明书操作手册文档范本/供应链核心企业金融数据分析与监控平台使用说明书V1.0.pdf`
- 新增说明书范本仅作为结构、页数、内容展开方式和产品个性化表达的参考，不得直接照抄正文

### 1.2 软件独立开发总原则

- 每一个任务都必须被视为一个全新的独立软件产品，禁止把历史任务当作“稍微改名后继续沿用”的基础版本
- 新任务的产品定位、功能骨架、角色分工、模块集合、路由结构、页面布局、业务数据、说明书正文和申请表文案都必须围绕当前任务重新组织
- 必须先根据当前产品标题、关键词和产品类型判断本次产品属于 `web` 端应用还是 `客户端` 应用，再分别生成匹配该形态的导航结构、操作壳层和页面布局
- 任务画像生成时必须优先输出项目专属 `project_dna`，至少覆盖 `preset_key`、`scene`、`focus_terms`、`module_signature`、`interaction_tone`、`visual_anchor`、`architecture_style`
- 行业命中不得再采用“首个关键词命中即返回”的粗放策略；必须按标题、关键词、行业做加权判定，避免把 `物流调度`、`供应链金融`、`供应链协同` 等不同产品误归到同一套模块骨架
- 除通用技术栈、基础框架和硬性规范外，不允许默认复用旧任务的业务主线、模块命名、页面结构和正文表达
- 任何任务在进入开发、重跑、补图、补文档前，都必须先确认当前产物是否确实属于该任务，不得混用旧任务的页面、截图、manifest、导出文档或说明书段落
- 若发现当前任务与历史任务在模块命名、页面结构、截图清单、正文表述上高度相似，必须优先判定为异常并立即回查画像、plan、build、capture 与文档生成链路
- 即使属于同一行业，不同产品也必须重新设计 UI/视觉效果，禁止统一沿用同一套导航配色、卡片层级、内容分区与模块骨架
- 不得在代码生成前向模型灌入固定的后台成品壳层，例如统一的“左侧深色竖栏导航 + 右侧内容区”框架；若当前产品需要类似结构，也必须由本次任务基于画像重新设计，而不是复用预置套板

### 1.3 任务独立性检查清单

- 开发前必须确认当前任务拥有独立的 `product_name`、`keyword`、`industry`、`user_roles`、`core_modules`、`required_pages`
- build 前必须确认任务专属画像、PRD 摘要、页面路由和截图场景均来自当前任务，不得引用旧任务残留文件
- capture 前必须清理旧 `screenshot_manifest.json`、旧失败截图和历史残留图片，防止串图
- publish 前必须确认说明书、源码文档、申请表和截图清单中的产品名称、模块标题、图注和正文均对应当前任务
- 验收时必须同时核对页面截图、说明书章节、申请表字段和源码文档目录，确认没有出现“旧任务页面 + 新任务封面”的混合状态

## 2. 软件说明书 / 操作手册规范

### 2.1 必须包含的章节

每份软件说明书 / 操作手册至少必须包含以下模块：

1. 封面
2. 文档说明
3. 引言
4. 软件概述
5. 开发设计 / 系统设计
6. 开发运行环境 / 软件适配环境
7. 功能结构说明
8. 软件使用说明
9. 技术特点说明

### 2.1.1 页数与内容丰富度硬标准

- 每份软件说明书 / 操作手册成品页数必须达到 20 页以上
- 不能只改封面标题或产品名称，正文内容必须围绕当前任务的关键词、产品名称、行业特征和模块结构重新生成
- 必须按当前任务生成新的模块说明、业务流程、角色权限、页面讲解和操作步骤，不得复用旧任务正文
- 在满足基础章节的前提下，应根据关键词和项目名称主动增加更贴合业务场景的模块，使不同软著说明书之间具备明显差异，避免千篇一律
- 每一个软件产品的设计说明、功能模块、页面讲解、业务流程、技术特点和截图说明都必须体现该软件产品自身的特色与个性，内容侧重点要随关键词、行业、角色和核心模块变化而变化
- 不得沿用历史任务的固定模板段落作为当前说明书主体内容；开发背景、开发目的、软件概述、功能结构说明、页面讲解和技术特点说明都必须根据当前任务画像重新拼装
- 若当前任务的标题、关键词、行业、角色或核心模块发生变化，则说明书正文对应段落也必须同步变化，不能出现“仅封面换名、主体仍是上一版”的情况

### 2.1.2 全新撰写硬标准

- 每一份说明书都必须视为“全新撰写”，禁止采用“上一份说明书替换标题后局部改字”的方式生成
- 说明书中的开发背景、开发目的、产品简介、功能结构说明、使用说明、技术特点、页面讲解、操作步骤、业务价值和图注都必须重新组织句式，不得整段复用旧任务正文
- 同一类行业产品允许保留少量行业共性表达，但必须确保当前任务的业务对象、角色动作、页面重点、结果产出和模块关系描述明显区别于历史任务
- 若两个任务的关键词、产品名称、行业、角色或模块不同，则说明书对应章节必须同步变化，且变化不能只停留在名词替换层面
- 页面截图说明必须严格引用当前截图中真实出现的页面标题、按钮、表头和业务要点撰写，不得沿用其他任务的页面说明
- 说明书中任何段落若能直接替换产品名后用于另一个任务，则该段落视为同质化内容，必须重写

### 2.2 引言最低要求

引言至少必须详细说明：

- 开发背景
- 开发目的
- 适用领域
- 适用对象 / 使用角色

### 2.3 开发设计 / 系统设计最低要求

开发设计 / 系统设计至少必须包含：

- 开发技术说明
- 开发语言说明
- 技术选型说明
- 系统总体架构说明
- 系统架构图
- 开发运行环境 / 软件适配环境
- 技术特性描述
- 软件核心功能 / 软件主要功能 / 功能元素说明

### 2.4 软件使用说明最低要求

软件使用说明必须尽量详细，至少覆盖：

- 登录方式
- 首页说明
- 所有主要页面的功能说明
- 所有主要页面的详细操作步骤
- 关键按钮、输入项、结果反馈说明
- 常见业务流程说明

### 2.5 页面截图要求

- 必须对软件所有主要页面进行真实截图
- 每个软件的页面截图数量必须达到 10 张以上
- 每一张截图都必须插入说明书
- 软件界面截图中的页面标题、导航标题、模块标题、区块标题和按钮文案必须使用中文，不允许再附带英文标题、英文副标题或英文路由标识
- 截图中可见的页面头部、左侧导航、模块卡片和标题区域应以中文为主，不得出现 `DASHBOARD`、`ASSETS`、`REPORTS`、`/assets` 这类英文标题型标识
- 每一张截图下都必须包含：
  - 图题 / 图注
  - 页面功能讲解
  - 详细操作说明
  - 页面要点说明
- 所有图片和页面中都不得再出现“页面摘要”区块
- 模块页面截图中的正文区不得出现大块空白；必须展示与当前业务主题直接相关的列表、卡片、统计信息、处理说明或其他有效业务内容
- 当真实数据不足时，页面正文区必须使用有业务意义的兜底内容填充，不能让主体区域呈现空白效果
- 页面文字示例、用户姓名、操作人姓名等业务数据必须使用真实风格姓名，不得使用 `张三`、`李四`、`王五`、`赵六` 这类占位式假名
- 页面元素说明、功能讲解、页面要点中的文本必须去除 emoji、图标字符和不可见符号，只保留可正常显示的正文文字
- 页面中的输入框、筛选条、按钮、标签、卡片标题和表格内容都必须显式使用稳定中文字体，不能在截图中出现方框字、空心框或丢失 glyph
- 同一产品内不同页面的内容模块必须真实区分，不能出现“只换标题、卡片标题和正文说明基本相同”的重复页面
- 内容区卡片与概览模块不得再使用“顶部一条高饱和蓝色粗边框”的效果，应采用完整边框、浅阴影或常规描边的正常 UI 效果
- 截图所对应的页面必须来自当前任务真实运行结果，不得使用旧任务图片替代，也不得使用“相似页面截图”冒充当前任务页面
- 截图图注、页面讲解和操作步骤必须与当前截图中的真实模块、真实按钮、真实表头和真实场景一致，禁止沿用旧任务的说明文字

### 2.6 禁止项

以下内容不得再出现在软件说明书 / 操作手册中：

- “文档中的截图均来自软件真实运行界面” 这句话
- “本次说明书基于真实运行页面自动采集……” 这类表述
- “以下内容根据当前任务产物自动整理，可作为申请表填写参考版本。” 这类申请表提示语
- “说明：本申请表为系统自动生成的填写稿……” 这类申请表说明语
- 整套 AI / 全自动开发流程相关描述
- 技术选型清单中的以下项目：
  - `UI组件库: Ant Design 5`
  - `浏览器自动化: Playwright`
  - `文档生成: python-docx`
- 任何模型厂商、模型名称或接口供应商名称，例如 `DeepSeek`、`OpenAI`、`Claude` 等
- 上述禁止项不包含通用技术框架、编程语言、数据库或工程工具名称；`FastAPI`、`React`、`TypeScript`、`PostgreSQL` 等技术说明允许保留英文原名
- `开发设计流程`
- `开发设计流程图`
- `部署维护说明`
- `部署说明`
- `运行维护说明`
- `数据与材料管理说明`
- `常见问题`
- 固定问答模板，例如“推荐使用哪种浏览器”“忘记密码怎么办”“说明书截图从哪里来”等 FAQ 句式

### 2.7 架构图要求

- 说明书中保留 `系统架构图`
- 标题与图注中产品名称和“系统架构图”之间不得出现多余空格，必须连贯排版
- 图中必须正确显示中文文字，不能出现大面积方框字
- 架构图必须使用真实可读的中文字体绘制
- 架构图必须生成标准图片并插入说明书，禁止再以文字版、ASCII 图、纯文本替代图形图片
- 图中每个方框都必须保证文字完整落在方框内部，不允许溢出到方框之外
- 架构图内容必须与实际软件结构一致
- 架构图不得继续使用所有任务共用的一套固定五层盒子布局；必须根据 `project_dna.architecture_style`、当前模块签名和业务主线生成项目专属布局、节点名称和图注
- 若两个任务的业务主线、模块签名或行业画像不同，则架构图中的布局结构、节点命名、颜色组合和图注表述也必须同步变化，不能只替换产品名称

### 2.8 排版要求

- 中文正文统一采用稳定的中文字体机制，避免字形上凸、下凸、错位
- 标题、正文、图注必须分别使用统一样式
- 不允许出现明显错位、乱码或中英文字体混排失衡
- 每一页都必须带有页脚页码
- 标题、图题、图注与模块名称必须避免不必要的断开和多余空格，保持连贯排版
- 句内不应出现无语义的空格，尤其产品名称与后续中文说明之间必须连续排版，例如不得出现“智慧园区管理平台 面向...”或“智慧园区管理平台 采用...”
- 除首页标题页外，说明书其余正文页必须尽量连续排版，不允许因为强制分页或宽松段距导致某页只有少量内容、剩余大面积留白
- 源代码文档所有页面都必须遵循紧凑排版原则，避免出现半页代码后下方整块空白

## 3. 源代码文档规范

### 3.1 单文档输出要求

每次只生成一份源代码文档：

1. 完整版：`source_code_book.docx`

### 3.2 完整版规则

- 必须收录纳入代码索引的全部源代码内容
- 必须按完整代码连续排版
- 不能截断成只保留部分页
- 不再生成“前 30 页 + 后 30 页”的提交版

### 3.3 代码页排版要求

- 每页代码必须尽量连续排版
- 不允许出现某一页只有少量代码、下方大段留白的情况
- 不允许因为不必要的强制分页造成大片空白
- 代码文档中的代码应尽量上下紧凑排版，减少由于空行、额外分隔或过宽行距造成的无效留白

### 3.3.1 代码注释控制要求

- 代码中的注释必须从严控制，只保留理解代码所必需的说明
- 如果只是纯说明性、重复代码表意、对运行逻辑没有帮助的文字注释，必须删除
- 不允许依靠注释堆砌来增加源码文档页数

### 3.4 代码体量要求

- 真实业务代码总量必须达到 60 页以上目标
- 不允许靠空白页、注释堆砌、重复样板代码凑页
- 应优先通过增加真实页面、真实接口、真实服务层、真实数据模型、真实业务逻辑来提升代码量

## 4. 生成机制要求

### 4.1 说明书生成机制

- 必须使用稳定的中文字体配置
- 必须对截图、图注、正文说明、操作步骤做统一排版
- 必须保证所有主要页面都能进入说明书
- 截图时必须确保页面主体内容完整落入图片范围，不允许出现右侧、下方或关键卡片超出截图边界的情况
- 页面右侧信息区、上半区概览卡片和说明卡片必须根据当前模块字段、状态、样例数据与业务关注点生成差异化内容，不能多个页面只换标题、主体说明基本相同
- 必须保证说明书正文、截图说明、模块说明和申请表字段均与当前任务一致，不能沿用历史任务旧内容
- 必须基于当前任务关键词、产品名、行业或 PRD 摘要生成任务专属画像，并用该画像驱动页面内容、截图场景、说明书章节和申请表内容
- 任务专属画像至少必须覆盖当前任务的 `product_name`、`keyword`、`industry`、`user_roles`、`core_modules` 与页面路由；说明书正文必须引用这些画像字段生成，而不是只替换标题变量
- 任务专属画像还必须生成当前产品的内容侧重点、典型场景、差异化模块表达和专属业务价值说明，用于保证不同产品的说明书与页面截图具有独立特色
- 任务专属画像还必须向下游 PRD、前端代码生成、页面截图、架构图与说明书 LLM 输入透传同一份 `project_dna` / `differentiation_hint`，不得只在画像层保存而不参与后续生成
- 任务专属画像必须同时给出当前产品的应用形态结论，即 `web` 端应用或 `客户端` 应用，并以此驱动导航方式、页面壳层和截图风格
- 任务专属画像中的 `experience_blueprint` 必须显式给出 `navigation_variant` 与 `shell_layout_hint`，并要求前端生成严格落实该壳层方向；当画像指定顶部导航、指挥条、分析画布、分段导航或协同工作区时，不得再退回统一左侧竖栏后台
- 前端核心页面 `App.tsx`、`Login.tsx`、`Dashboard.tsx` 与全部模块页必须由当前任务的 DeepSeek 调用全新生成；本地仅允许保留运行基础设施、依赖、字体和画像数据，不得预先写入固定页面骨架
- 若 DeepSeek 未生成核心前端页面、生成结果无效，build 必须直接失败并重新生成，不得再从本地模板回填 `App.tsx`、首页、登录页或模块页
- 传给 DeepSeek 的输入应以规则、PRD、任务书、任务画像和模块约束为主，不得把固定后台布局成品、固定导航壳层或通用页面套板作为隐含前置实现塞给模型
- 当 `core_modules` 或页面路由发生变化时，模块说明、角色说明、业务流程、截图讲解、表格示例和页面要点也必须随之变化
- 文档说明模块中不得再写入“本次说明书基于真实运行页面自动采集……”这类固定句式
- 说明书章节内容必须主动扩写，确保每份成品不少于 `20` 页；不能依靠放大字号、增加空白或重复语句凑页
- 说明书、图注、架构图、环境说明和技术说明中不得出现模型名称、模型厂商名称或接口供应商名称，尤其不得出现 `DeepSeek`
- 技术说明中的框架、语言、数据库和工程工具名称应保留行业常用英文原名，不必强制改写为中文
- 页面和代码生成阶段的 DeepSeek v4 提示词必须显式要求“为当前产品重新设计 UI，不复用统一后台套板，不复用统一模块骨架，不复用统一视觉配色”
- 说明书 LLM 提示词或模板提示中必须显式要求“不得复用历史任务正文结构、不得只换标题、不得输出通用后台套板段落”
- 若核心模块、路由、角色或页面布局被修正，说明书生成前必须重新读取最新 `project_profile`、最新截图清单和最新 PRD 摘要，禁止继续使用旧版中间产物撰写正文

### 4.1.1 软件独立开发流程

1. 先读取当前任务关键词、产品名、行业、PRD 摘要和历史修复规则，建立当前任务专属画像
2. 基于当前任务专属画像生成模块集合、角色集合、页面路由、截图场景和文档重点
3. 生成页面代码时先验证核心页、模块页和导航是否体现当前任务；若 DeepSeek 结果缺失或不合格，必须直接判失败并重新生成，禁止回退到本地固定模板
4. 截图前验证运行页标题、导航、模块名称和产品标识是否属于当前任务，发现错页必须中止 capture
5. 文档生成前重新读取当前任务的最新截图清单、页面标题、模块说明和申请表字段，确保正文写作基于最新产物
6. 发布前进行交叉验收，确认页面、截图、说明书、源码文档、申请表与当前任务完全一致后再导出

### 4.2 图示生成机制

- 系统架构图必须使用支持中文的字体绘制
- 图内文字必须真实可读
- 不得输出大面积方框字
- 若中文字体缺失，应优先使用项目内置中文字体文件补齐，不得回退为文本版架构图
- 系统架构图导出结果必须为标准图片文件并直接插入说明书，不得输出“系统架构图（文本版）”
- 必须具备“文字适配方框”的自动机制；当文字较长时，应自动换行、缩放字体或增大方框，保证文字不越界

### 4.3 源码导出机制

- 只允许导出单份完整源码文档 `source_code_book.docx`
- 不再保留 `submission` 导出模式
- 必须避免末页多余分页
- 必须保证代码页按固定行数连续排版
- 源码文档收录的源码必须与当前任务实际生成的软件内容一致，不能混入旧模板项目的主体业务代码
- 源码文档排版必须以“减少留白、提高有效代码密度”为原则
- 真实业务代码总页数必须稳定达到并尽量超过 `60` 页

### 4.4 标准交付清单

- 每个任务完成后，下载列表必须至少提供以下 4 份文件：
  1. 软件说明书 / 操作手册
  2. 软件源代码
  3. 全套 ZIP 压缩文件
  4. 申请表
- 任务详情页中的页面截图和工作列表必须保留，用于查看执行进度和产物质量
- 全套 ZIP 必须包含当前任务的软件源码、工作区、PRD/清单、截图工件、导出文档和申请表

### 4.5 申请表生成要求

- 申请表必须作为标准导出文件参与发布与下载
- 申请表至少必须填写以下字段：
  - 软件名称
  - 版本号
  - 软件简称
  - 开发完成日期
  - 软件分类
  - 开发的硬件环境
  - 运行的硬件环境
  - 开发该软件的操作系统
  - 源程序量
  - 软件开发环境 / 开发工具
  - 该软件的运行平台 / 操作系统
  - 软件运行支撑环境 / 支持软件
  - 编程语言
  - 开发目的
  - 面向领域 / 行业
  - 软件的主要功能
  - 软件的技术特点
- 其中：
  - 开发目的不得少于 100 字
  - 面向领域 / 行业不得少于 4 字
  - 软件的主要功能不得少于 500 字
  - 软件的技术特点不得少于 100 字

## 5. 验收口径

若出现以下任一情况，则视为不达标：

- 说明书缺少引言 / 系统设计 / 软件使用说明
- 说明书缺少开发语言说明或技术选型说明
- 说明书缺少系统架构图
- 说明书中出现被禁止的 AI / 自动开发流程描述
- 说明书页数不足 20 页
- 页面截图少于 10 张
- 说明书截图没有全部插入
- 某张截图没有详细说明
- 任一页面缺少页脚页码
- 文档说明中仍出现“本次说明书基于真实运行页面自动采集……”这类固定句式
- 任一图片或页面中仍出现“页面摘要”
- 任一模块页正文区仍呈现明显空白、缺少有效业务内容
- 新任务导出的说明书、源码文档或申请表正文仍沿用旧任务主要内容
- 新任务的软件模块集合、路由结构、页面布局或截图清单与历史任务高度重复，且无明确业务原因
- 当前产品形态本应为客户端却仍生成成普通 Web 后台，或本应为 Web 应用却套用客户端壳层
- 当前任务的 `project_profile`、截图清单、说明书正文、申请表字段之间相互不一致
- 说明书章节、页面图注或操作说明中出现可直接套用于其他任务的大段通用后台表述
- 页面截图、说明书图片或导出文档混入旧任务文件、旧任务模块标题或旧任务产品标识
- 说明书中仍出现 `部署维护说明`、`部署说明`、`运行维护说明`、`数据与材料管理说明` 或 `常见问题` 等固定章节
- 两个不同任务的说明书仅替换产品名称或封面标题，正文主体段落、模块说明和页面讲解仍高度重复
- 说明书正文、功能讲解或页面要点中仍出现占位式假名
- 说明书正文、功能讲解或页面要点中仍出现无法正常显示的 emoji / 图标字符
- 说明书正文、图注、架构图或环境说明中仍出现 `DeepSeek` 等模型名称或供应商名称
- 说明书技术说明无故把 `FastAPI`、`React`、`TypeScript` 等通用技术原名改成不规范的中文替代写法
- 软件界面截图中仍出现英文标题、英文副标题或英文路由标识
- 软件界面截图中的输入框、按钮、标签或中部工具条仍出现方框字
- 同一产品内多个页面仍是同一块内容模块反复复用，仅更换标题
- 软件内容区仍保留顶部高饱和蓝色粗边框效果
- 架构图文字显示为方框
- 架构图仍以文本版或 ASCII 图形式出现在说明书中
- 架构图标题存在多余空格，或方框内文字溢出
- 源代码文档不是单份完整源码文档
- 下载列表缺少 ZIP 或申请表
- 源代码文档分页留白严重
- 源代码文档中仍存在大量无必要纯注释
- 真实业务代码总页数不足 60 页

## 6. 执行原则

- 本规范优先级高于单次任务中的默认模板
- 若现有生成逻辑与本规范冲突，必须修改生成逻辑以满足本规范
- 后续所有项目生成均默认继承本规范
