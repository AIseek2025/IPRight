# IPRight 文档生成制作规范

## 1. 适用范围

本规范用于约束 `IPRight` 后续每一次生成的软件说明书 / 操作手册、源代码文档及其相关导出物。

所有后续生成任务均必须遵循本规范，不得因单次项目差异而随意放宽。

## 2. 软件说明书 / 操作手册规范

### 2.1 必须包含的章节

每份软件说明书 / 操作手册至少必须包含以下模块：

1. 封面
2. 文档说明
3. 引言
4. 软件概述
5. 开发设计 / 系统设计
6. 开发运行环境 / 软件适配环境
7. 功能结构说明
8. 软件使用说明
9. 技术特点说明
10. 常见问题

### 2.2 引言最低要求

引言至少必须详细说明：

- 开发背景
- 开发目的
- 适用领域
- 适用对象 / 使用角色

### 2.3 开发设计 / 系统设计最低要求

开发设计 / 系统设计至少必须包含：

- 开发技术说明
- 开发语言说明
- 技术选型说明
- 系统总体架构说明
- 系统架构图
- 开发运行环境 / 软件适配环境
- 技术特性描述
- 软件核心功能 / 软件主要功能 / 功能元素说明

### 2.4 软件使用说明最低要求

软件使用说明必须尽量详细，至少覆盖：

- 登录方式
- 首页说明
- 所有主要页面的功能说明
- 所有主要页面的详细操作步骤
- 关键按钮、输入项、结果反馈说明
- 常见业务流程说明

### 2.5 页面截图要求

- 必须对软件所有主要页面进行真实截图
- 每一张截图都必须插入说明书
- 每一张截图下都必须包含：
  - 图题 / 图注
  - 页面功能讲解
  - 详细操作说明
  - 页面要点说明
- 页面文字示例、用户姓名、操作人姓名等业务数据必须使用真实风格姓名，不得使用 `张三`、`李四`、`王五`、`赵六` 这类占位式假名
- 页面元素说明、功能讲解、页面要点中的文本必须去除 emoji、图标字符和不可见符号，只保留可正常显示的正文文字

### 2.6 禁止项

以下内容不得再出现在软件说明书 / 操作手册中：

- “文档中的截图均来自软件真实运行界面” 这句话
- 整套 AI / 全自动开发流程相关描述
- 技术选型清单中的以下项目：
  - `UI组件库: Ant Design 5`
  - `浏览器自动化: Playwright`
  - `文档生成: python-docx`
  - `AI模型: DeepSeek v4-pro / v4-flash`
- `开发设计流程`
- `开发设计流程图`

### 2.7 架构图要求

- 说明书中保留 `系统架构图`
- 标题与图注中产品名称和“系统架构图”之间不得出现多余空格，必须连贯排版
- 图中必须正确显示中文文字，不能出现大面积方框字
- 架构图必须使用真实可读的中文字体绘制
- 图中每个方框都必须保证文字完整落在方框内部，不允许溢出到方框之外
- 架构图内容必须与实际软件结构一致

### 2.8 排版要求

- 中文正文统一采用稳定的中文字体机制，避免字形上凸、下凸、错位
- 标题、正文、图注必须分别使用统一样式
- 不允许出现明显错位、乱码或中英文字体混排失衡
- 标题、图题、图注与模块名称必须避免不必要的断开和多余空格，保持连贯排版
- 句内不应出现无语义的空格，尤其产品名称与后续中文说明之间必须连续排版，例如不得出现“智慧园区管理平台 面向...”或“智慧园区管理平台 采用...”

## 3. 源代码文档规范

### 3.1 单文档输出要求

每次只生成一份源代码文档：

1. 完整版：`source_code_book.docx`

### 3.2 完整版规则

- 必须收录纳入代码索引的全部源代码内容
- 必须按完整代码连续排版
- 不能截断成只保留部分页
- 不再生成“前 30 页 + 后 30 页”的提交版

### 3.3 代码页排版要求

- 每页代码必须尽量连续排版
- 不允许出现某一页只有少量代码、下方大段留白的情况
- 不允许因为不必要的强制分页造成大片空白

### 3.4 代码体量要求

- 真实业务代码总量必须达到 60 页以上目标
- 不允许靠空白页、注释堆砌、重复样板代码凑页
- 应优先通过增加真实页面、真实接口、真实服务层、真实数据模型、真实业务逻辑来提升代码量

## 4. 生成机制要求

### 4.1 说明书生成机制

- 必须使用稳定的中文字体配置
- 必须对截图、图注、正文说明、操作步骤做统一排版
- 必须保证所有主要页面都能进入说明书

### 4.2 图示生成机制

- 系统架构图必须使用支持中文的字体绘制
- 图内文字必须真实可读
- 不得输出大面积方框字
- 必须具备“文字适配方框”的自动机制；当文字较长时，应自动换行、缩放字体或增大方框，保证文字不越界

### 4.3 源码导出机制

- 只允许导出单份完整源码文档 `source_code_book.docx`
- 不再保留 `submission` 导出模式
- 必须避免末页多余分页
- 必须保证代码页按固定行数连续排版

## 5. 验收口径

若出现以下任一情况，则视为不达标：

- 说明书缺少引言 / 系统设计 / 软件使用说明
- 说明书缺少开发语言说明或技术选型说明
- 说明书缺少系统架构图
- 说明书中出现被禁止的 AI / 自动开发流程描述
- 说明书截图没有全部插入
- 某张截图没有详细说明
- 说明书正文、功能讲解或页面要点中仍出现占位式假名
- 说明书正文、功能讲解或页面要点中仍出现无法正常显示的 emoji / 图标字符
- 架构图文字显示为方框
- 架构图标题存在多余空格，或方框内文字溢出
- 源代码文档不是单份完整源码文档
- 源代码文档分页留白严重
- 真实业务代码总页数不足 60 页

## 6. 执行原则

- 本规范优先级高于单次任务中的默认模板
- 若现有生成逻辑与本规范冲突，必须修改生成逻辑以满足本规范
- 后续所有项目生成均默认继承本规范
